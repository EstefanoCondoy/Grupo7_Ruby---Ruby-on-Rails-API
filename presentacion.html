Contexto del Proyecto

Objetivos del Proyecto

Migracion Tecnologica

Migrar el backend de E-Commerce de Spring Boot (Java) a Ruby on Rails 7.1 en modo API-only. Rails aplica el principio de "Convention over Configuration", lo que reduce drasticamente la cantidad de archivos de configuracion y acelera el desarrollo. Se implemento el patron MVC con una capa adicional de Services para separar la logica de negocio de los controladores.

Seguridad Moderna

Implementar autenticacion sin estado (stateless) con JWT (JSON Web Tokens) firmados con el algoritmo HS256 y expiracion configurable de 24 horas. Las contrasenas se cifran con BCrypt, que genera un hash con salt aleatorio. Nunca se almacenan en texto plano ni se exponen en las respuestas JSON gracias a los Serializers (DTOs).

Entorno Replicable

Contenerizar la API y la base de datos PostgreSQL con Docker y Docker Compose. Cualquier persona puede clonar el repositorio y ejecutar un unico comando (docker-compose up) para tener la aplicacion corriendo sin instalar Ruby, PostgreSQL ni ninguna gema manualmente. Elimina el clasico problema de "en mi maquina si funciona".

Controllers (app/controllers/api/v1/): Reciben peticiones HTTP en formato JSON. Validan parametros con Strong Parameters y delegan toda la logica a los Services. No contienen logica de negocio, siguiendo el principio de "controladores delgados".
  • Services (app/services/): AuthService genera y decodifica tokens JWT. ProductService maneja busqueda y paginacion. ReceiptService gestiona transacciones atomicas, validacion de stock y calculo de totales. UserService maneja registro y autenticacion.
  • Models (app/models/): Definen relaciones ActiveRecord (has_many, belongs_to, has_secure_password), validaciones de datos (presence, numericality, uniqueness, format) y metodos de dominio como sufficient_stock?(qty) y deduct_stock!(qty).
  • Serializers (app/serializers/): Actuan como DTOs (Data Transfer Objects). Transforman modelos Ruby a JSON con formato camelCase. Aseguran que campos sensibles como password_digest nunca se expongan al cliente.
  • ExceptionHandler (app/controllers/concerns/): Concern centralizado que intercepta 8 tipos de excepciones: AuthenticationError, TokenExpired, InvalidToken, ValidationError, InsufficientStockError, RecordNotFound, RecordInvalid y ParameterMissing. Todas devuelven JSON estandarizado.
Base de Datos

Modelado de Datos — 4 Migraciones

User has_many :receipts

idfirst_namelast_nameemailpassword_digest

Contrasena cifrada con BCrypt mediante has_secure_password. Email unico con validacion case-insensitive y normalizado a minusculas con callback before_save. Al eliminar un usuario, se eliminan sus recibos en cascada (dependent: :destroy). Validaciones de longitud maxima de 50 caracteres.

Product has_many :receipt_items

idnamedescriptionprice (decimal 10,2)stock (integer)

Precio con tipo decimal(10,2) para precision monetaria sin errores de redondeo. Stock validado como entero mayor o igual a 0. Metodo sufficient_stock?(qty) verifica disponibilidad antes de la compra. Metodo deduct_stock!(qty) descuenta de forma segura y lanza error si es insuficiente.

Receipt belongs_to :user

iduser_id (FK)total (decimal 10,2)

El campo "total" se calcula EXCLUSIVAMENTE en el backend; el cliente NO lo envia. Esto previene que un usuario malicioso manipule el monto. El total es la suma de los subtotales de sus ReceiptItems. has_many :receipt_items con dependent: :destroy para eliminar items en cascada.

ReceiptItem Tabla Pivote (N:M)

idreceipt_id (FK)product_id (FK)quantityunit_pricesubtotal

Tabla intermedia que conecta Receipt con Product (relacion muchos a muchos). unit_price se toma del precio ACTUAL del catalogo al momento de la compra. subtotal = unit_price * quantity. Esto permite mantener un historial de precios aunque el catalogo cambie despues.

Acceso Protegido

El cliente envia el header: Authorization: Bearer <token>. El metodo authorize_request de ApplicationController lo decodifica, identifica al usuario y permite o rechaza el acceso.

# ApplicationController — Intercepta cada peticion protegida
def authorize_request
  header = request.headers['Authorization']
  raise InvalidToken, 'Token no proporcionado' unless header
  token = header.split(' ').last
  decoded = AuthService.decode(token) # Valida firma HS256 + expiracion
  @current_user = User.find(decoded[:user_id])
end
# Si el token es invalido o expirado -> ExceptionHandler retorna 401 Unauthorized

Logica de Negocio

Reglas de Negocio

1

Total calculado exclusivamente en el backend

El cliente solo envia user_id e items [{product_id, quantity}]. El servidor obtiene los precios directamente del catalogo en la base de datos, calcula cada subtotal (unit_price * quantity) y la suma total. Esto impide que un atacante manipule los precios interceptando la peticion HTTP. Si el cliente enviara el total, podria cambiarlo a $0.01.

2

Validacion y descuento atomico de stock

Antes de confirmar el recibo, se verifica que haya stock suficiente con sufficient_stock?(qty). Al confirmarse, se descuenta con deduct_stock!(qty). Se usa Product.lock (SELECT FOR UPDATE de PostgreSQL, bloqueo pesimista) para evitar condiciones de carrera cuando dos clientes compran el mismo producto simultaneamente. Todo dentro de una transaccion con rollback automatico.

3

Contrasenas cifradas y ocultas en las respuestas

BCrypt genera un hash con salt aleatorio usando has_secure_password de Rails. Los Serializers nunca incluyen password_digest en las respuestas JSON. El email se normaliza a minusculas con before_save para evitar duplicados por variacion de mayusculas (ej: User@Mail.com vs user@mail.com).

4

Manejo centralizado de errores (ExceptionHandler)

Todas las respuestas de error (400 Bad Request, 401 Unauthorized, 404 Not Found, 409 Conflict, 422 Unprocessable Entity) siguen un formato JSON identico: { success: false, error: "tipo", message: "detalle", status: codigo, timestamp: "ISO 8601" }. Se interceptan 8 tipos de excepciones personalizadas.

Implementacion

Codigo Clave: ReceiptService

El corazon del sistema. Todo esta envuelto en una transaccion atomica de ActiveRecord: si cualquier paso falla, TODOS los cambios se revierten automaticamente (rollback).

class ReceiptService
  def self.create(user_id, items_params)
    user = User.find(user_id)
    raise ValidationError, 'El recibo debe contener al menos un item' if items_params.blank?

    ActiveRecord::Base.transaction do # ── Inicio de transaccion atomica ──
      receipt = Receipt.create!(user: user, total: 0)
      total = BigDecimal('0') # Precision monetaria exacta

      items_params.each do |item|
        product = Product.lock.find(item[:product_id]) # Bloqueo pesimista (SELECT FOR UPDATE)
        quantity = item[:quantity].to_i

        # (1) Validar que hay stock suficiente
        unless product.sufficient_stock?(quantity)
          raise InsufficientStockError, "Stock insuficiente para '#{product.name}'"
        end

        # (2) Crear item con precio del CATALOGO, no del cliente
        ReceiptItem.create!(receipt: receipt, product: product,
          quantity: quantity, unit_price: product.price,
          subtotal: product.price * quantity)

        product.deduct_stock!(quantity) # (3) Descontar stock automaticamente
        total += product.price * quantity
      end

      receipt.update!(total: total) # (4) Total = suma de todos los subtotales
    end # ── Si algo falla, ROLLBACK total automatico ──
  end
end

Dockerfile: Imagen base ruby:3.2.2-slim. Instala dependencias del sistema (build-essential, libpq-dev, curl). Copia el Gemfile, ejecuta bundle install con 4 workers paralelos y 3 reintentos, luego copia el codigo fuente.
  • docker-compose.yml: Orquesta dos servicios (api y db). Define un healthcheck para PostgreSQL (pg_isready -U postgres cada 5s). La API espera a que la BD este saludable antes de arrancar (depends_on: condition: service_healthy).
  • Volumenes persistentes: postgres_data persiste la base de datos entre reinicios de contenedores. bundle_cache cachea las gemas Ruby para que los builds posteriores sean mucho mas rapidos.
  • Comando de arranque: rm -f tmp/pids/server.pid (limpia PIDs huerfanos) + rails db:prepare (crea o migra la BD) + rails db:seed (carga datos de prueba) + rails server en 0.0.0.0:3000.
  • Variables de entorno: DB_HOST, DB_PORT, DB_USERNAME, DB_PASSWORD, JWT_SECRET_KEY, JWT_EXPIRATION_HOURS, RAILS_ENV. Todas configurables via .env o docker-compose.yml.

GET /products?page=1&per_page=10 retorna metadata con currentPage, totalPages y totalCount. Configurable por el cliente para optimizar el rendimiento.

Busqueda por Nombre

Swagger UI (OpenAPI 3.0)

Documentacion interactiva en /api-docs generada con Rswag. Visualiza parametros, esquemas de request/response y permite probar endpoints en vivo.

Postman Collection

17 requests con scripts post-response. Al hacer Login, un script automatico guarda el token JWT como variable de coleccion. No hay que copiar/pegar tokens.

Seeds (Datos de Prueba)

db/seeds.rb crea 4 usuarios (los integrantes + admin) y 10 productos reales con precios y stock. Se ejecuta automaticamente al levantar Docker.

ExceptionHandler

8 tipos de excepciones interceptadas. Formato JSON estandarizado para todos los errores: success, error, message, status y timestamp ISO 8601.

Serializers (DTOs)

Transforman modelos ActiveRecord a JSON con camelCase. Ocultan password_digest y formatean las respuestas. Equivalentes a los DTOs de Spring Boot.

Transacciones Atomicas

ReceiptService usa ActiveRecord::Base.transaction + Product.lock (bloqueo pesimista). Si falla cualquier paso, rollback automatico total sin datos inconsistentes.
